## 1. PURPOSE, SCOPE, AND HOW TO USE DO-178C

### Purpose (§1.1)
DO-178C provides guidance for producing software for airborne systems and equipment that performs its intended function with a level of confidence in safety that complies with airworthiness requirements. It provides:
- Objectives for software life cycle processes
- Activities that provide a means for satisfying those objectives
- Descriptions of evidence (software life cycle data) indicating objectives are satisfied
- Variations in objectives by software level
- Additional considerations (COTS, PDS, tool qualification, alternative methods)

### Scope (§1.2)
Covers software in airborne systems used on aircraft, engines, propellers, and (by region) APUs. Does NOT address: operational aspects, firmware classification (must be treated as HW or SW), organizational structure, personnel qualification, commercial relationships. A complete description of the system life cycle or certification process is not intended.

### Key Usage Rules (§1.4)
- Not mandated by law — represents aviation community consensus
- "Shall" and "must" are deliberately avoided; guidance uses "should"
- If adopted as means of compliance: ALL applicable objectives must be satisfied
- Supplements exist for specific techniques (DO-330: tools, DO-331: MBD, DO-332: OOT, DO-333: FM) — if a supplement exists for a technique being used, it MUST be used
- Compliance = all applicable objectives satisfied by performing all planned activities with evidence captured
- Tables in Annex A are NOT a checklist — full body must be considered

### Relationship to DO-178B
Major changes in DO-178C vs DO-178B:
- Explicit supplement structure (DO-330, DO-331, DO-332, DO-333)
- Data coupling and control coupling added as Level A objectives (§6.4.4.2.d/e)
- Parameter data items formalized (§2.5.1, §6.6, §11.22)
- Clarifications on structural coverage on object code (§6.4.4.2.b)
- Deactivated code categories formally defined (§6.4.4.3.d)
- Tool qualification TQL levels clarified (Table 12-1)

## 2. SYSTEM ASPECTS RELATING TO SOFTWARE DEVELOPMENT (§2)

### System Requirements Allocation to Software (§2.1)
System requirements allocated to software may include:
- Functional and operational requirements (§2.1.a)
- Interface requirements (§2.1.b)
- Performance requirements (§2.1.c)
- Safety-related requirements including safety strategies, design constraints, partitioning, dissimilarity, redundancy, safety monitoring (§2.1.d)
- Security requirements (§2.1.e)
- Maintenance requirements (§2.1.f)
- Certification requirements (§2.1.g)
- Additional requirements to aid system life cycle (§2.1.h)

### Information Flow — System to Software (§2.2.1)
Data passed from system processes to software processes:
- System requirements allocated to software
- System safety objectives
- Software level for software components + description of associated failure condition(s)
- System description and hardware definition
- Design constraints (external interfaces, partitioning requirements)
- Details of system activities to be performed as part of software life cycle
- Evidence of acceptability of data provided by software processes

### Information Flow — Software to System (§2.2.2)
Data passed from software to system processes:
- Details of **derived requirements** created during software life cycle — CRITICAL for safety assessment
- Description of software architecture including partitioning
- Evidence of system activities performed by software life cycle
- Problem/change documentation, identified HW/SW incompatibilities
- Any limitations of use
- Configuration identification and status constraints
- Performance, timing, accuracy characteristics

### System Safety Assessment and Software Level (§2.3)

#### Failure Condition Categorization — Table 2-1
| Category | Description |
|----------|-------------|
| **Catastrophic** | Multiple fatalities, usually with loss of airplane |
| **Hazardous** | Large reduction in safety margins; physical distress/excessive workload preventing accurate crew task performance; serious/fatal injury to small number of occupants |
| **Major** | Significant reduction in safety margins/functional capabilities; significant crew workload increase; discomfort/injuries to passengers |
| **Minor** | No significant aircraft safety reduction; crew actions well within capabilities; slight safety margin reduction; slight workload increase |
| **No Safety Effect** | No effect on safety; no effect on operational capability or pilot workload |

#### Software Level Definition (§2.3.3)
| Level | Contribution to Failure Condition |
|-------|----------------------------------|
| **A** | Anomalous behavior would cause or contribute to a CATASTROPHIC failure condition |
| **B** | Anomalous behavior would cause or contribute to a HAZARDOUS failure condition |
| **C** | Anomalous behavior would cause or contribute to a MAJOR failure condition |
| **D** | Anomalous behavior would cause or contribute to a MINOR failure condition |
| **E** | Anomalous behavior would have NO EFFECT — no DO-178C objectives apply (must be confirmed by certification authority) |

#### Software Level Determination Key Rules (§2.3.4)
- Level is based on CONTRIBUTION to failure condition — not the system-level level directly
- Development to a software level does NOT imply assignment of failure rate — software reliability rates cannot be used in safety assessment like hardware failure rates
- If anomalous behavior contributes to MULTIPLE failure conditions → assign the level associated with the MOST SEVERE
- If partitioning between software components cannot be demonstrated → all components assigned the most severe level
- Software Level for each component must be AGREED WITH CERTIFICATION AUTHORITY

### Architectural Considerations (§2.4)

#### Partitioning (§2.4.1)
Partitioning provides isolation between software components to contain/isolate faults. Rules:
- A partitioned component SHALL NOT contaminate another's code, I/O, or data storage (§2.4.1.a)
- May consume shared processor resources ONLY during scheduled period of execution (§2.4.1.b)
- Failures of hardware unique to one partitioned component shall not adversely affect others (§2.4.1.c)
- Software providing partitioning must have SAME OR HIGHER software level as the highest level of any partitioned component (§2.4.1.d) — critical rule
- Hardware providing partitioning must be assessed by system safety assessment (§2.4.1.e)
- Only PARTITIONED components can be assigned individual software levels

#### Multiple-Version Dissimilar Software (§2.4.2)
- Produces two or more software components performing the same function via different means
- Also called: multi-version software, dissimilar software, N-version programming, software diversity
- Lifecycle processes completed BEFORE dissimilarity is introduced remain potential error sources
- Degree of dissimilarity is generally NOT measurable
- Usually used as ADDITIONAL protection AFTER normal DO-178C verification objectives are satisfied
- Verification methods may be reduced from single-version requirements (per §12.3.2)

#### Safety Monitoring (§2.4.3)
Safety monitoring protects against specific failure conditions by directly monitoring a function. Three key attributes of a monitor (§2.4.3):
- **Software level**: Monitoring software assigned the level of the MOST SEVERE failure condition of the monitored function
- **System fault coverage**: Monitor design must detect faults under ALL necessary conditions
- **Independence**: Monitor and protective mechanism cannot be rendered inoperative by the same failure that causes the failure condition

### Software Considerations in System Life Cycle Processes (§2.5)

#### Parameter Data Items (§2.5.1)
A data set that influences software behavior without modifying EOC and is managed as a separate configuration item. Examples: configuration tables, databases. Parameter data items may:
- Influence paths executed through executable object code
- Activate or deactivate software components/functions
- Adapt computations to system configuration
- Provide computational data
- Establish time and memory partitioning allotments
- Provide initial values

**Key rule:** Parameter data items must be assigned the SAME software level as the software component using them. See §6.6 for verification guidance.

#### User-Modifiable Software (§2.5.2)
A user-modifiable component can be changed by the user without certification authority review, IF:
- User modification shall not adversely affect safety, operational capabilities, flight crew workload, non-modifiable components, or protection mechanisms
- System requirements must include provision for user modification
- Software protecting the non-modifiable component must be at the SAME software level as the non-modifiable component
- At time of modification, user takes responsibility for all CM, QA, and verification of the modifiable software

#### COTS Software (§2.5.3)
COTS software must satisfy the objectives of DO-178C. If software life cycle data is deficient, it must be augmented. See §12.1.4 (Upgrading a Development Baseline) and §12.3.4 (Product Service History).

#### Option-Selectable Software (§2.5.4)
Functions selected by software-programmed options rather than hardware connector pins. A means must be provided to prevent inadvertent selections of non-approved configurations. Deactivated code guidance applies (§5.2.4, §6.4.4.3.d).

#### Field-Loadable Software (§2.5.5)
Software loadable without removing the system from installation. Safety considerations:
- Detection of corrupted or partially loaded software
- Effects of loading inappropriate software
- HW/SW compatibility; SW/SW compatibility; aircraft/SW compatibility
- Inadvertent enabling of field-loading function
- Loss or corruption of software configuration identification display

**Key rule:** Unless justified by system safety assessment, the detection mechanism for partial/corrupted loads must be assigned the SAME failure condition or software level as the most severe function using that load.

## 3. SOFTWARE LIFE CYCLE (§3)

### Software Life Cycle Processes (§3.1)
Three categories:
1. **Software planning process** (§4) — defines and coordinates development and integral processes
2. **Software development processes** (§5) — requirements, design, coding, integration
3. **Integral processes** (§6–§9) — verification, CM, QA, certification liaison — performed CONCURRENTLY throughout the life cycle (not just at the end)

### Software Life Cycle Definition (§3.2)
Project defines one or more software life cycles by:
- Choosing activities for each process
- Specifying a sequence for activities
- Assigning responsibilities

The life cycle may be iterative (entered/re-entered). Different components of the same product can use different development sequences. Prototyping is recognized (used to refine requirements before final development).

### Transition Criteria (§3.3)
"Transition criteria" (NOT "exit criteria") determine whether a process may be entered or re-entered. Key points:
- Not every input need be complete before a process can be initiated, IF transition criteria are satisfied
- If a process acts on partial inputs, subsequent inputs must be examined to confirm previous outputs are still valid
- Transition criteria may include: verification reviews completed, input is an identified configuration item, traceability analysis completed

## 4. SOFTWARE PLANNING PROCESS (§4)

### Planning Process Objectives (§4.1)
The planning process must define:
- Activities of development and integral processes to address system requirements and software level(s) (§4.1.a)
- The software life cycle(s) including inter-relationships, sequencing, feedback mechanisms, and transition criteria (§4.1.b)
- The software life cycle environment including methods and tools for each process (§4.1.c)
- Additional considerations (§4.1.d)
- Software development standards consistent with system safety objectives (§4.1.e)
- Software plans complying with §4.3 and §11 (§4.1.f)
- Development and revision coordination of software plans (§4.1.g)

### Software Plans (§4.3)
The five required plans:
| Plan | Section | Primary Purpose |
|------|---------|----------------|
| **PSAC** | §11.1 | Primary means of communicating proposed development to certification authority; defines means of compliance |
| **SDP** | §11.2 | Defines software life cycle(s), development environment, how development process objectives satisfied |
| **SVP** | §11.3 | Defines how verification process objectives satisfied |
| **SCMP** | §11.4 | Defines how CM process objectives satisfied |
| **SQAP** | §11.5 | Defines how QA process objectives satisfied |

Plans must define transition criteria specifying:
- Inputs to the process (including feedback)
- Integral process activities required on those inputs
- Availability of tools, methods, plans, and procedures

### Software Life Cycle Environment Planning (§4.4)

#### Software Development Environment (§4.4.1)
- Development environment can adversely affect software (compiler errors, linker failures to detect memory allocation errors)
- Tool combinations should be chosen so an error introduced by one would be detected by another
- Tool qualification need must be assessed (see §12.2)
- Optional features of software tools must be examined — especially important for compilers and autocode generators
- Known tool problems and limitations must be assessed

#### Language and Compiler Considerations (§4.4.2)
- Compilers with optimization features: if test cases give coverage consistent with software level, correctness of optimization need not be separately verified
- Some compilers produce object code not directly traceable to source code (initialization, built-in error detection, exception handling) — means to detect this must be planned
- **If a new compiler version, linkage editor, or loader version is introduced, or compiler options change → previous tests and coverage analyses may NO LONGER BE VALID**
- Compiler acceptable only for that specific product once verification objectives satisfied — not automatically acceptable for other products

#### Software Test Environment (§4.4.3)
- Emulator or simulator may need to be qualified (§12.2)
- Differences between target computer and emulator/simulator must be considered and documented in SVP

### Software Development Standards (§4.5)
Three types required:
- **Software Requirements Standards** (§11.6) — methods, rules, tools for HLR development
- **Software Design Standards** (§11.7) — methods, rules, tools for architecture and LLR development
- **Software Code Standards** (§11.8) — programming languages, methods, rules, tools for coding

Standards must:
- Enable uniform design and implementation across components
- Disallow constructs/methods producing unverifiable outputs or incompatible with safety requirements
- Consider robustness
- Consider defensive programming practices

## 5. SOFTWARE DEVELOPMENT PROCESSES (§5)

### Overview
Development processes produce one or more levels of software requirements:
- **High-Level Requirements (HLRs)**: produced directly from system requirements and system architecture
- **Low-Level Requirements (LLRs)**: produced from HLRs and architecture during software design process; sufficient detail to directly implement source code without further information
- If source code is generated directly from HLRs → HLRs are ALSO considered LLRs and all LLR guidance applies

**Derived requirements** arise from design/implementation decisions not directly traceable to higher-level requirements. Examples:
- Interrupt handling software for the target computer
- Periodic monitor iteration rate not specified by system requirements
- Scaling limits for fixed point arithmetic

ALL derived requirements must be made available to system processes including the system safety assessment process — they may introduce safety-significant failure modes not captured in the FHA/PSSA.

### Software Requirements Process (§5.1)

#### Objectives (§5.1.1)
- HLRs are developed (§5.1.1.a)
- Derived HLRs are defined and provided to system processes including safety assessment (§5.1.1.b)

#### Activities (§5.1.2)
Key activities:
- Analyze system requirements for ambiguities, inconsistencies, undefined conditions (§5.1.2.a)
- Report inadequate/incorrect inputs as feedback (§5.1.2.b)
- Each system requirement allocated to software must be specified in HLRs (§5.1.2.c)
- HLRs must address system requirements to preclude system hazards (§5.1.2.d)
- HLRs must conform to Software Requirements Standards, be verifiable and consistent (§5.1.2.e)
- HLRs must be stated in quantitative terms with tolerances where applicable (§5.1.2.f)
- HLRs must NOT describe design or verification detail except for justified design constraints (§5.1.2.g)
- Derived HLRs and reasons for their existence must be defined (§5.1.2.h)
- Derived HLRs must be provided to system processes including safety assessment (§5.1.2.i)

### Software Design Process (§5.2)

#### Objectives (§5.2.1)
- Software architecture and LLRs developed from HLRs (§5.2.1.a)
- Derived LLRs defined and provided to system processes (§5.2.1.b)

#### Activities (§5.2.2)
Key activities:
- LLRs and architecture must conform to Software Design Standards, be traceable, verifiable, consistent
- Derived LLRs and rationale must be defined and analyzed to ensure higher-level requirements not compromised
- Design process may INTRODUCE or PRECLUDE failure modes → additional data defined as derived requirements and provided to system processes
- Interfaces (data flow, control flow) must be consistent between components
- Control flow and data flow must be monitored when safety-related requirements dictate (watchdog timers, reasonableness-checks, cross-channel comparisons)
- **Complexity must be avoided** — as complexity increases, it becomes more difficult to verify the design and satisfy safety requirements (Note at §5.2.2)

#### Designing for Deactivated Code (§5.2.4)
Deactivated code = code that cannot be executed in a specific configuration (unselected functionality, unused library functions). Differs from DEAD code.
- A mechanism must be designed to assure deactivated code has no adverse impact on active code
- Evidence must be available that deactivated code is disabled in environments where not intended
- Development of deactivated code must comply with DO-178C objectives just as active code

### Software Coding Process (§5.3)

#### Objective (§5.3.1)
Source Code is developed from low-level requirements.

#### Activities (§5.3.2)
- Source Code must implement LLRs and conform to software architecture (§5.3.2.a)
- Source Code must conform to Software Code Standards (§5.3.2.b)
- Autocode generator use must conform to constraints defined in planning (§5.3.2.d)
- **PATCHES**: Shall not be used in software submitted for certification to implement changes in requirements, architecture, or changes found necessary from verification. Patches may be used on limited, case-by-case basis only for known deficiencies in the development environment (e.g., known compiler problem). If used, must have: CM tracking confirmation, analysis showing patched SW satisfies all objectives, justification in SAS (§5.4.2.e/f)

### Integration Process (§5.4)

#### Objective (§5.4.1)
EOC and associated Parameter Data Item Files, if any, are produced and loaded into the target hardware for hardware/software integration.

#### Activities (§5.4.2)
- Object code and EOC generated from Source Code and compiling/linking/loading data
- Software integration may be on host computer, target emulator, or target computer
- Software loaded into target computer for HW/SW integration
- Compiling, linking, and loading data becomes part of lifecycle data

### Software Development Process Traceability (§5.5)
Bi-directional trace data required:
- **System requirements ↔ HLRs** (§5.5.a): enables verification of complete implementation; gives visibility to derived HLRs
- **HLRs ↔ LLRs** (§5.5.b): enables verification of complete HLR implementation; gives visibility to derived LLRs and architectural decisions
- **LLRs ↔ Source Code** (§5.5.c): enables verification no undocumented function is implemented; enables verification of complete LLR implementation

## 6. SOFTWARE VERIFICATION PROCESS (§6)

### Critical Principle (§6.0)
**"Verification is not simply testing. Testing, in general, cannot show the absence of errors."**

The software verification process is a combination of reviews, analyses, and tests. The verification process detects and reports errors; REMOVAL of errors is an activity of the SOFTWARE DEVELOPMENT PROCESSES.

### Purpose of Software Verification (§6.1)
The verification process verifies that:
- System requirements allocated to software → developed into HLRs satisfying those requirements (§6.1.a)
- HLRs → developed into architecture and LLRs satisfying the HLRs (§6.1.b)
- Architecture and LLRs → developed into Source Code satisfying LLRs and architecture (§6.1.c)
- EOC satisfies software requirements (intended function) AND provides confidence in absence of unintended functionality (§6.1.d)
- EOC is robust — responds correctly to abnormal inputs and conditions (§6.1.e)
- Means used to perform verification are technically correct and complete for the software level (§6.1.f)

### Verification Independence (§6.2)
Independence = verification performed by a person(s) other than the developer of the item being verified. A tool may achieve equivalence to human verification. **The person who created the test cases should not be the same person who developed the Source Code from those LLRs.**

Reverification is required after corrective actions and/or changes.

### Software Reviews and Analyses (§6.3)

**Distinction:** Reviews = qualitative assessment. Analyses = repeatable evidence of correctness.
A review may use a checklist. An analysis examines in detail functionality, performance, traceability, safety implications.

#### Review of HLRs (§6.3.1) — Objectives confirmed:
- Compliance with system requirements — functions defined, functional/performance/safety requirements satisfied, derived requirements correctly defined
- Accuracy and consistency — each HLR accurate, unambiguous, sufficiently detailed, no conflicts
- Compatibility with target computer — no conflicts with HW/SW features, response times, I/O hardware
- Verifiability — each HLR can be verified
- Conformance to Software Requirements Standards — standards followed, deviations justified
- Traceability — functional/performance/safety requirements from system developed into HLRs
- Algorithm aspects — accuracy and behavior of algorithms, especially discontinuities

#### Review of LLRs (§6.3.2) — Objectives confirmed:
- Compliance with HLRs — LLRs satisfy HLRs; derived requirements and design basis correctly defined
- Accuracy and consistency — each LLR accurate, unambiguous, no conflicts
- Compatibility with target computer — bus loading, response times, I/O hardware
- Verifiability
- Conformance to Software Design Standards
- Traceability — HLRs and derived requirements developed into LLRs
- Algorithm aspects

#### Review of Software Architecture (§6.3.3) — Objectives confirmed:
- Compatibility with HLRs — no conflict with HLRs especially partitioning schemes
- Consistency — correct relationships between components via data flow and control flow; if interface is to lower-level component, higher-level component must have appropriate protection mechanisms
- Compatibility with target computer — especially initialization, asynchronous operation, synchronization, interrupts
- Verifiability — no unbounded recursive algorithms
- Conformance to Software Design Standards
- Partitioning integrity — partitioning breaches prevented

#### Review of Source Code (§6.3.4) — Objectives confirmed:
- Compliance with LLRs — Source Code accurate and complete; no undocumented function implemented
- Compliance with software architecture — data flow and control flow match architecture definition
- Verifiability — no statements/structures that cannot be verified; code does not need to be altered to test it
- Conformance to Software Code Standards — complexity restrictions, code constraints; deviations justified
- Traceability — LLRs developed into Source Code
- Accuracy and consistency — includes stack usage, memory usage, fixed point arithmetic overflow, floating-point arithmetic, resource contention, worst-case execution timing, exception handling, use of uninitialized variables, cache management, unused variables, data corruption due to task or interrupt conflicts; impact of compiler and linker options on worst-case execution timing must be assessed

#### Review of Integration Process Outputs (§6.3.5)
Ensures outputs of integration process are complete and correct. Examines compiling, linking, loading data, and memory map. Typical potential errors: compiler warnings, incorrect hardware addresses, memory overlaps, missing software components.

### Software Testing (§6.4)

**Testing confirms the EOC:**
- Complies with HLRs (§6.4.a)
- Is robust with HLRs (§6.4.b)
- Complies with LLRs (§6.4.c)
- Is robust with LLRs (§6.4.d)
- Is compatible with the target computer (§6.4.e)

**Three testing types:**
- **HW/SW Integration Testing** (§6.4.3.a): target computer environment; errors with interrupt handling, execution time, HW transients, data bus contention, built-in test failures, HW/SW interface errors, control loop behavior, memory management, stack overflow, partitioning violations
- **Software Integration Testing** (§6.4.3.b): inter-relationships between SW requirements, implementation of requirements by SW architecture; errors include incorrect initialization, parameter passing errors, data corruption (global data), inadequate end-to-end numerical resolution, incorrect event sequencing
- **Low-Level Testing** (§6.4.3.c): compliance of each component with its LLRs; errors include algorithm failures, incorrect loop operations, incorrect logic decisions, failure to process legitimate input combinations, incorrect responses to missing/corrupted inputs, exception handling failures, incorrect computation sequence, inadequate algorithm precision

**Note:** If test cases for HW/SW integration or SW integration satisfy both requirements-based coverage AND structural coverage → not necessary to duplicate for low-level testing.

### Requirements-Based Test Selection (§6.4.2)

#### Normal Range Test Cases (§6.4.2.1)
- Real and integer input variables exercised using valid equivalence classes AND boundary values
- Time-related functions (filters, integrators, delays): multiple code iterations to check characteristics
- State transitions: test cases for transitions possible during normal operation
- Logic equations: test cases verifying variable usage and Boolean operators

#### Robustness Test Cases (§6.4.2.2)
Demonstrate software responds correctly to ABNORMAL inputs and conditions:
- Real/integer variables exercised with invalid equivalence class values
- System initialization during abnormal conditions
- Failure modes of incoming data, especially complex digital data strings
- Out-of-range loop count computation attempts
- Protection mechanisms for exceeded frame times
- Arithmetic overflow protection for time-related functions
- Invalid state transitions (not allowed by requirements)

### Test Coverage Analysis (§6.4.4)

Four objectives for test coverage analysis:
- Test coverage of HLRs achieved (§6.4.4.a)
- Test coverage of LLRs achieved (§6.4.4.b)
- Test coverage of software structure to appropriate coverage criteria achieved (§6.4.4.c)
- Test coverage of software structure — data coupling AND control coupling achieved (§6.4.4.d)

#### Requirements-Based Test Coverage Analysis (§6.4.4.1)
- Analysis using Trace Data to confirm test cases exist for EACH software requirement
- Analysis to confirm test cases satisfy normal AND robustness testing criteria
- Resolution of deficiencies: add/enhance test cases
- Analysis to confirm ALL test cases used for structural coverage are traceable to requirements

#### Structural Coverage Analysis (§6.4.4.2)
Determines which code structure was NOT exercised by requirements-based test procedures.

**Coverage levels by software level:**

| Coverage Criterion | Required at Level | Definition |
|-------------------|------------------|------------|
| **Statement Coverage** | A, B, C | Every statement executed at least once |
| **Decision Coverage** | A, B, C | Every Boolean decision evaluated to both True and False |
| **MC/DC** | A, B | Every condition in a decision independently shown to affect outcome |
| **Data Coupling** | A | Data interactions between components at interfaces verified |
| **Control Coupling** | A | Control flow interactions between components verified |

**MC/DC — Modified Condition/Decision Coverage (§6.4.4.2.c):**  
Every condition in a decision has been shown to INDEPENDENTLY affect the decision outcome. Requires:
1. Every entry/exit point invoked
2. Every condition in a decision takes every possible outcome (True and False)
3. Each condition independently shown to affect the decision outcome — one condition varies while all others remain fixed, and the decision outcome changes

For N conditions in a decision: N+1 test cases typically sufficient for MC/DC (compared to 2^N for exhaustive testing). **Masking**: when a condition's value cannot affect the decision outcome due to short-circuit evaluation or other conditions — MC/DC requires unmasking each condition.

**Structural coverage may be performed on Source Code, object code, or EOC (§6.4.4.2.b).** If software level is A and compiler/linker generates additional code not directly traceable to Source Code statements → additional verification must establish correctness of such generated code sequences. Note: compiler-generated array-bound checks are NOT considered directly traceable to Source Code and require additional verification.

**Data Coupling (§6.4.4.2.d — Level A only):** Confirms requirements-based testing has exercised data interactions between code components at their interfaces. Addresses passed parameters, global data, interface errors.

**Control Coupling (§6.4.4.2.e — Level A only):** Confirms requirements-based testing has exercised control flow interactions between code components. Addresses correct sequencing and conditionality of calls.

#### Structural Coverage Analysis Resolution (§6.4.4.3)
When code structure is not exercised, causes and resolutions:
- **Shortcomings in test cases/procedures (§6.4.4.3.a):** Augment test cases or change test procedures. May need to review requirements-based coverage analysis methods.
- **Inadequacies in software requirements (§6.4.4.3.b):** Modify software requirements and develop additional test cases.
- **Extraneous code including dead code (§6.4.4.3.c):** Remove code. Analyze effect and need for reverification. Dead code found at Source Code or object code level MAY remain only if analysis shows it does not exist in EOC (e.g., smart compiling). Procedures must prevent inclusion in future builds.
- **Deactivated code (§6.4.4.3.d):** Two categories:
  - *Category One*: Not intended to be executed in ANY current certified configuration. Combination of analysis and testing shows means by which it could inadvertently execute are prevented, isolated, or eliminated. Any software level reassignment justified by system safety assessment and documented in PSAC.
  - *Category Two*: Only executed in certain approved configurations. Operational configuration for execution must be established; additional test cases developed to satisfy required coverage.

**Dead code vs Deactivated code distinction:**
- Dead code = unreachable code, cannot be executed regardless of configuration
- Deactivated code = intentionally inactive in some configurations but potentially active in others

### Verification of Parameter Data Items (§6.6)
May be verified SEPARATELY from EOC verification if ALL of the following apply:
- EOC developed and verified by normal range testing to correctly handle all Parameter Data Item Files
- EOC is robust with respect to Parameter Data Item File structures and attributes
- All behavior of EOC resulting from parameter data item contents can be verified
- Structure of lifecycle data allows parameter data item to be managed separately

If verified separately:
- Parameter Data Item File verified to comply with its structure as defined by HLRs
- Each data element shown to have correct value, be consistent with other elements, comply with attributes
- All elements have been covered during verification

## 7. SOFTWARE CONFIGURATION MANAGEMENT PROCESS (§7)

### SCM Process Objectives (§7.1)
- Each configuration item and successive versions labeled unambiguously (§7.1.a)
- Baselines defined for further lifecycle activity (§7.1.b)
- Problem reporting records non-compliance, deficiencies, anomalous behavior; ensures resolution (§7.1.c)
- Change control provides recording, evaluation, resolution, and approval of changes (§7.1.d)
- Change review ensures problems and changes are assessed, approved/disapproved, implemented; feedback provided (§7.1.e)
- Status accounting provides configuration management data (§7.1.f)
- Archival and retrieval ensures software lifecycle data can be retrieved for duplication, regeneration, retest, or modification (§7.1.g)
- Software load control ensures EOC and Parameter Data Items loaded with appropriate safeguards (§7.1.h)
- Software lifecycle environment control ensures tools are identified, controlled, and retrievable (§7.1.i)

**SCM objectives are INDEPENDENT of software level.** However two control categories exist (CC1 and CC2).

### SCM Activities (§7.2)

Key sub-processes:
- **Configuration Identification (§7.2.1)**: Establish identification for all lifecycle data; identification established BEFORE item used by other processes, referenced by other data, or used for manufacture/loading
- **Baselines and Traceability (§7.2.2)**: Baselines in controlled software libraries protected from change; change control activities required to develop derivative baselines; traceability to baseline from which derived
- **Problem Reporting, Tracking, and Corrective Action (§7.2.3)**: Problem Report describing process non-compliance, output deficiency, or software anomalous behavior; PR must invoke change control activity for corrective action
- **Change Control (§7.2.4)**: Changes traced to origin; lifecycle processes repeated from point at which change affects their outputs; lifecycle data updated; change control is aided by change review
- **Change Review (§7.2.5)**: Impact assessment on system requirements, feedback to system processes; impact on lifecycle data; confirmation of configuration identification; feedback to affected processes
- **Configuration Status Accounting (§7.2.6)**: Reporting on CI identification, baseline ID, PR status, change history, release status
- **Archive, Retrieval, and Release (§7.2.7)**: Integrity procedures (no unauthorized changes, appropriate media, duplicate copies in separate archives); configuration items identified and released before use for manufacture; minimum: EOC and Parameter Data Item Files, if any

### Data Control Categories (§7.3)
**CC1 (Control Category 1)**: Full set of SCM activities — configuration identification, baselines, traceability, problem reporting, full change control (tracking, records), change review, status accounting, full archival (retrieval, protection, media selection, release, data retention)

**CC2 (Control Category 2)**: Subset — configuration identification only, retrieval, protection against unauthorized changes, data retention (NO baselines, NO problem reporting, NO full change control tracking)

**Which data gets which category:** Specified in Annex A tables by software level. Higher-criticality lifecycle data gets CC1; supporting data may get CC2 at lower levels.

### Software Load Control (§7.4)
Procedures for part numbering and media identification to ensure only approved software is loaded.

### Software Lifecycle Environment Control (§7.5)
Configuration identification for executable code (or equivalent) of tools. SCM for qualified tools: per CC1 or CC2 as directed by §12.2.3. Non-qualified build/load tools (compilers, assemblers, linkage editors): minimum CC2.

- Assurance that plans and standards are developed and reviewed for compliance and consistency

- Software conformity review prior to delivery (§8.3)

- Problem Reports evaluated with status recorded

## 11. SOFTWARE LIFE CYCLE DATA (§11)

### Characteristics of All Lifecycle Data
All software lifecycle data must be:
1. **Unambiguous** — only one interpretation possible
2. **Complete** — includes necessary requirements/descriptive material; all figures labeled; terms defined
3. **Verifiable** — can be checked for correctness by a person or tool
4. **Consistent** — no conflicts
5. **Modifiable** — structured for changes without breaking the structure
6. **Traceable** — origin of components can be determined

### §11.1 — Plan for Software Aspects of Certification (PSAC)
Primary document for certification authority to assess proposed software lifecycle. Must include:
- **System overview**: functions and allocation to HW/SW, architecture, processors, HW/SW interfaces, safety features
- **Software overview**: software functions emphasizing safety and partitioning concepts (resource sharing, redundancy, fault tolerance, single event upset mitigation, timing/scheduling)
- **Certification considerations**: certification basis, means of compliance, proposed software level(s), justification from safety assessment, potential failure condition contributions
- **Software life cycle**: software life cycle definition; summary of each process; how objectives satisfied; organizations involved; responsibilities; system lifecycle and certification liaison responsibilities
- **Software life cycle data**: data to be produced and controlled; relationship between data items; data submitted to authority; form of data; means to make data available
- **Schedule**: how applicant will provide visibility of lifecycle process activities for authority review planning
- **Additional considerations**: alternative methods, tool qualification, previously developed software, option-selectable, user-modifiable, deactivated code, COTS, field-loadable, parameter data items, dissimilar software, product service history
- **Supplier oversight**: means of ensuring supplier processes and outputs comply with plans and standards

### §11.2 — Software Development Plan (SDP)
May be included in PSAC. Must include:
- Standards: identification of requirements, design, and code standards
- Software life cycle: detailed description of lifecycle processes including transition criteria
- Software development environment: requirements development methods/tools, design methods/tools, coding methods/languages/tools (including autocode generator options/constraints), compilers/linkage editors/loaders, hardware platforms for tools

### §11.3 — Software Verification Plan (SVP)
Must include:
- Organization and independence: organizational responsibilities; methods for establishing verification independence when required
- Verification methods: review methods (checklists); analysis methods (traceability, coverage analysis); testing methods (test case selection, test procedures, test data)
- Verification environment: test equipment, testing and analysis tools; differences from target computer
- Transition criteria for entering verification
- Partitioning considerations if partitioning used
- Compiler assumptions
- Reverification method for software modification
- Previously developed software considerations if applicable
- Multiple-version dissimilar software if applicable

### §11.4 — Software Configuration Management Plan (SCMP)
Must include: environment, configuration identification methods, baselines/traceability procedures, problem reporting details, change control procedures, change review methods, configuration status accounting, archive/retrieval/release, software load control, lifecycle environment controls, lifecycle data controls (CC1/CC2), supplier control.

### §11.5 — Software Quality Assurance Plan (SQAP)
Must include: environment/scope/organizational responsibilities/standards/procedures/tools; authority/independence statement; activities for each lifecycle process; transition criteria; timing of SQA activities; SQA Records definition; supplier oversight.

### §11.6 — Software Requirements Standards
Methods, rules, tools for HLR development. Includes: requirements development methods; notation; tool use constraints; method for providing derived requirements to system processes.

### §11.7 — Software Design Standards
Methods, rules, tools for architecture and LLR development. Includes: design description methods; naming conventions; conditions on scheduling, interrupts, event-driven architectures, dynamic tasking, re-entry, global data, exception handling; design tool constraints; design constraints (e.g., exclusion of recursion, dynamic objects, data aliases, compacted expressions); complexity restrictions (maximum nesting, unconditional branches, entry/exit points).

### §11.8 — Software Code Standards
Programming language methods, rules, tools. Includes: language(s) and defined subset(s); source code presentation standards (line length, indentation, documentation); naming conventions; conditions on coupling and complexity of expressions; coding tool constraints.

### §11.9 — Software Requirements Data (SRD)
Definition of HLRs including derived requirements. Content:
- Allocation of system requirements to software with attention to safety-related requirements and potential failure conditions
- Functional and operational requirements per mode of operation
- Performance criteria (precision, accuracy)
- Timing requirements and constraints
- Memory size constraints
- HW and SW interfaces (protocols, formats, frequency of inputs/outputs)
- Failure detection and safety monitoring requirements
- Partitioning requirements; how partitioned components interact; software level(s) of each partition

### §11.10 — Design Description
Definition of software architecture and LLRs that satisfy HLRs. Content:
- Detailed description of how software satisfies HLRs (algorithms, data structures, allocation to processors/tasks)
- Software architecture defining structure
- Input/output description (data dictionary)
- Data flow and control flow
- Resource limitations (timing, memory margins, methods for measuring)
- Scheduling procedures and inter-processor/inter-task communication (time-rigid sequencing, preemptive scheduling, interrupts)
- Design methods details (software loading, user-modifiable, dissimilar software)
- Partitioning methods and means of preventing breaches
- Descriptions of components (new or previously developed; if PDS, reference to baseline)
- Derived requirements from software design process
- If deactivated code: description of means to ensure code cannot be enabled in target computer
- Rationale for design decisions traceable to safety-related system requirements

### §11.11 — Source Code
Code written in source language(s). For each component: software identification including name, date of revision/version.

### §11.12 — Executable Object Code (EOC)
Form of code directly usable by the target computer's processing unit — the software actually loaded into hardware.

### §11.13 — Software Verification Cases and Procedures
Details how verification process activities are implemented. Content:
- Review and analysis procedures: scope and depth of review/analysis methods
- Test cases: purpose of each test case, inputs, conditions, expected results to achieve required coverage, pass/fail criteria
- Test procedures: step-by-step instructions for setup, execution, results evaluation, test environment

### §11.14 — Software Verification Results
Produced by verification process activities. Must:
- For each review, analysis, and test: indicate each procedure that passed/failed and final pass/fail results
- Identify the CI or software version reviewed, analyzed, or tested
- Include results of tests, reviews, analyses, coverage analyses, and traceability analyses
- Any discrepancies found must be recorded and tracked via problem reporting

### §11.15 — Software Life Cycle Environment Configuration Index (SECI)
Identifies configuration of the software lifecycle environment. Aids reproduction of environment for regeneration, reverification, or modification. Must identify:
- Hardware and its operating system software
- Tools used during development (compilers, linkage editors, loaders, data integrity tools, autocode generators with options)
- Test environment (testing and analysis tools)
- Qualified tools and their qualification data

### §11.16 — Software Configuration Index (SCI)
Identifies configuration of the software product. Must identify:
- Software product
- EOC and Parameter Data Item Files
- Each Source Code component
- Previously developed software in the product
- Software lifecycle data
- Archive and release media
- Build instructions (compiling, linking instructions; recovery procedures for regeneration/testing/modification)
- Reference to SECI (if separate)
- Data integrity checks for EOC (if used)
- Procedures for modifying user-modifiable software (if any)
- Procedures for loading software into target hardware

### §11.17 — Problem Reports
Must include:
- Identification of CI and/or lifecycle process activity in which problem observed
- Identification of CI(s) to be modified or process to be changed
- Problem description with sufficient detail to assess potential safety/functional effects
- Description of corrective action taken to resolve the problem

### §11.18 — Software Configuration Management Records
Results of SCM process activities. Examples: CI identification lists, baseline/software library records, change history reports, archive records, release records.

### §11.19 — Software Quality Assurance Records
Results of SQA process activities. May include: SQA review/audit reports, meeting minutes, records of authorized process deviations, software conformity review records.

### §11.20 — Software Accomplishment Summary (SAS)
**Primary data item for showing compliance with the PSAC.** The closure document submitted at certification. Must include:
- **System overview**: similar to PSAC with differences noted
- **Software overview**: safety and partitioning concepts used; differences from PSAC proposal
- **Certification considerations**: restates and describes differences from PSAC
- **Software life cycle**: actual lifecycle(s) used; differences from PSAC
- **Software life cycle data**: differences from PSAC proposals; references to SCI and SECI by configuration identifiers and version
- **Additional considerations**: specific issues requiring certification authority attention; differences from PSAC
- **Supplier oversight**: how supplier processes and outputs comply
- **Software identification**: software configuration by part number and version
- **Software characteristics**: EOC size, timing margins including worst-case execution time, memory margins, resource limitations, means used for measuring each characteristic
- **Change history**: summary of changes with attention to safety-affecting failures; improvements to lifecycle processes since previous certification
- **Software status**: summary of OPEN Problem Reports at time of certification. For each open PR: description, associated errors, functional limitations, operational restrictions, potential adverse safety effects, justification for allowing PR to remain open, mitigation actions
- **Compliance statement**: statement of compliance with DO-178C; summary of methods used to demonstrate compliance; deviations from plans/standards/DO-178C not covered elsewhere

### §11.21 — Trace Data
Bi-directional associations between lifecycle data items:
- System requirements allocated to software ↔ HLRs
- HLRs ↔ LLRs
- LLRs ↔ Source Code
- Software requirements ↔ test cases
- Test cases ↔ test procedures
- Test procedures ↔ test results

### §11.22 — Parameter Data Item File
Form of data directly usable by target computer's processing unit. For each instantiation: reference to SAS of associated EOC if packaged separately.

## 12. ADDITIONAL CONSIDERATIONS (§12)

### Use of Previously Developed Software (§12.1)

#### Modifications to PDS (§12.1.1)
Activities:
- Revised system safety assessment outputs must be reviewed
- If software level revised: upgrading development baseline guidance (§12.1.4)
- Both requirement changes AND architecture changes must be analyzed for impact
- Affected area must be determined via data flow, control flow, timing, traceability analysis
- Affected areas reverified per §6

#### Change of Aircraft Installation (§12.1.2)
System safety assessment determines if software level and certification basis are the same for new installation. If same → no additional effort required. If functional modifications required → §12.1.1 guidance. If previous development didn't produce required outputs → §12.1.4 guidance.

#### Change of Application or Development Environment (§12.1.3)
If different compiler or compiler options → different object code → previous test results may NOT be valid for new application. If different processor → change impact analysis required including: new/modified components, previous HW/SW integration test results applicable, additional HW/SW integration tests needed.

#### Upgrading a Development Baseline (§12.1.4)
Applies to: COTS software, airborne software developed to other guidance, software developed before DO-178C existence, software previously developed to DO-178C at lower software level.
- Objectives of DO-178C must be satisfied while taking advantage of previous lifecycle data
- Software life cycle data from previous development evaluated to ensure verification process objectives satisfied for new application
- Reverse engineering may be used to regenerate missing lifecycle data
- Strategy for compliance must be specified in PSAC

### Tool Qualification (§12.2)

#### Determining if Tool Qualification is Needed (§12.2.1)
**Qualification of a tool is needed when processes of DO-178C are ELIMINATED, REDUCED, OR AUTOMATED by use of a software tool WITHOUT its output being verified as specified in §6.**

The purpose: ensure tool provides confidence at least equivalent to the process(es) eliminated, reduced, or automated.

A tool is qualified only for use on a SPECIFIC SYSTEM where intention to use is stated in the PSAC. If previously qualified on one system → must be RE-QUALIFIED for another system.

#### Tool Qualification Level (TQL) Determination — Table 12-1 (§12.2.2)
Three criteria for assessing tool impact:
- **Criteria 1**: Tool output is part of airborne software and thus could INSERT an error (development tool)
- **Criteria 2**: Tool automates verification process(es) and could FAIL TO DETECT an error, AND output used to justify ELIMINATION or REDUCTION of other verification/development processes (verification tool removing other verification)
- **Criteria 3**: Tool that within its intended use could FAIL TO DETECT an error (verification tool, errors caught by other means)

TQL assignment (apply criteria sequentially, 1 before 2 before 3):

| Software Level | Criteria 1 | Criteria 2 | Criteria 3 |
|---------------|------------|------------|------------|
| A | TQL-1 | TQL-4 | TQL-5 |
| B | TQL-2 | TQL-4 | TQL-5 |
| C | TQL-3 | TQL-5 | TQL-5 |
| D | TQL-4 | TQL-5 | TQL-5 |

TQL-1 = most rigorous; TQL-5 = least rigorous.

**Examples:**
- Autocode generator (output IS airborne software) at Level A → Criteria 1 → TQL-1
- VectorCAST (coverage tool, output eliminates structural coverage analysis) at Level A → Criteria 2 → TQL-4
- VectorCAST (output reviewed manually by engineer) → no qualification needed (output verified per §6)
- Compiler at Level A → Criteria 1 → TQL-1 (typically handled by showing compiler "acceptable" after all verification objectives satisfied, per §4.4.2)

#### Tool Qualification Process (§12.2.3)
Objectives, activities, guidance and lifecycle data for each TQL → defined in DO-330, "Software Tool Qualification Considerations."

### Alternative Methods (§12.3)

#### Exhaustive Input Testing (§12.3.1)
If software component is simple and isolated with bounded input/output space → may substitute for one or more verification activities in §6. Must define complete set of valid inputs/outputs, analyze isolation, develop rationale, develop test cases.

#### Multiple-Version Dissimilar Software Verification (§12.3.2)
Methods for producing multiple dissimilar versions:
- Source Code implemented in two or more different programming languages
- Object code generated using different compilers
- Each version executes on separate, dissimilar processor or single processor with partitioning between versions
- SW requirements, design, and/or code developed by two or more development teams with managed interactions
- Development in two or more environments; each version verified using separate test environments
- Linked/loaded using different linkage editors and loaders
- Conformance to different Software Requirements Standards, Design Standards, and/or Code Standards

Additional objectives for dissimilar software:
- Inter-version compatibility requirements satisfied (normal/abnormal operations, state transitions)
- Equivalent error detection achieved

**Independence of dissimilar software (§12.3.2.1):** Applicant must demonstrate different teams with limited interaction developed each version's requirements, design, and Source Code. Independent test coverage analyses still required.

#### Software Reliability Models (§12.3.3)
DO-178C DOES NOT provide guidance for software reliability models based on developmental metrics — at time of writing, methods did not provide results in which confidence could be placed.

#### Product Service History (§12.3.4)
Certification credit may be granted if equivalent safety demonstrated via product service history. Acceptability depends on: CM of software, effectiveness of problem reporting, stability/maturity of software, relevance and length of service history, actual error rates, impact of modifications. Use, conditions, and results must be defined and submitted to certification authority.

## 14. DEEP Q&A BANK — DO-178C INTERVIEW QUESTIONS

**Q1: What is the fundamental difference between DO-178C and DO-178B?**
A: DO-178C (2011) introduced a formal supplement architecture not present in DO-178B (1992). Four supplements address specific techniques: DO-330 (tool qualification), DO-331 (model-based development/design), DO-332 (object-oriented technology), DO-333 (formal methods). If a supplement exists for a technique being used, that supplement MUST be used alongside DO-178C. DO-178C also added data coupling and control coupling as explicit Level A objectives (§6.4.4.2.d/e), formalized parameter data items (§2.5.1 and §11.22), and provided clarifications on structural coverage of compiler-generated code (§6.4.4.2.b). Terminology was also harmonized with ARP4754A.

**Q2: Explain the five software levels in DO-178C and what determines the level assignment.**
A: DO-178C §2.3.3 defines five software levels (A through E) based on the severity of the failure condition the software can contribute to, as determined by the system safety assessment process. Level A maps to Catastrophic failure conditions, Level B to Hazardous/Severe-Major, Level C to Major, Level D to Minor, and Level E to No Safety Effect. Key rule: if anomalous behavior of a software component contributes to multiple failure conditions, the component is assigned the level associated with the MOST SEVERE condition (§2.3.4). The level must be AGREED WITH THE CERTIFICATION AUTHORITY. Development to a software level does not imply any specific failure rate — software reliability rates cannot be used in the safety assessment the same way as hardware failure rates.

**Q3: What is the difference between a High-Level Requirement and a Low-Level Requirement?**
A: High-Level Requirements (HLRs, §5.1) are derived from system requirements and the system architecture. They define what the software must do — functional, performance, interface, and safety-related requirements at a level that can be verified but does not yet prescribe the implementation. They must be verifiable, unambiguous, and stated in quantitative terms with tolerances. HLRs must NOT describe design or verification detail except for justified design constraints. Low-Level Requirements (LLRs, §5.2) are derived from HLRs and the software architecture during the design process. They define how the software does it — in sufficient detail to implement Source Code DIRECTLY without further information. LLRs include timing, memory constraints, interface definitions, algorithm details, data structures. Both require independent review at Levels A and B.

**Q4: What is a derived requirement and why is it critical from a safety perspective?**
A: A derived requirement (§5.0, §5.1.1.b, §5.2.1.b) is a requirement that arises from a design or implementation decision rather than being directly traceable to a higher-level requirement. Examples from DO-178C: need for interrupt handling software for the target computer; specification of a periodic monitor's iteration rate not specified by system requirements; addition of scaling limits for fixed point arithmetic. They are critical because: (1) they represent functionality NOT captured in the system requirements, so they may introduce unintended behavior or failure modes that weren't analyzed in the FHA/PSSA; and (2) they MUST be made available to the system processes including the system safety assessment process so the safety assessment can be updated. Failure to identify and report derived requirements is a common certification finding and represents a safety risk.

**Q5: Explain verification independence in DO-178C. When is it required?**
A: Verification independence (§6.2) means the verification activity is performed by a person(s) other than the developer of the item being verified. Critically: it does NOT require a separate company or organization — it requires different personnel. A tool may achieve equivalence to human verification for independence purposes.

Independence is required at different levels for different activities:
- **Level A**: Independence required for reviews of HLRs, LLRs, architecture, Source Code, AND independence required for test case development (person creating test cases shall not be the same who developed the Source Code from those LLRs) AND independent test execution
- **Level B**: Independence required for reviews of HLRs, LLRs, architecture, Source Code, and test case review; test execution independence not explicitly required
- **Level C**: Reduced independence requirements
- **Level D**: Minimal independence requirements

**Q6: Describe all five types of structural coverage and state which software levels require each.**
A: DO-178C §6.4.4.2 defines structural coverage:
- **Statement Coverage**: Every statement in the code executed at least once. Required at Levels A, B, C.
- **Decision Coverage**: Every Boolean decision (condition or expression controlling flow) evaluates to both True and False at least once. Required at Levels A, B, C.
- **MC/DC (Modified Condition/Decision Coverage)**: Every condition in a decision has been shown to independently affect the decision outcome. For a decision with N conditions: typically N+1 test cases. Required at Levels A and B.
- **Data Coupling**: Data interactions between code components at interfaces are exercised by requirements-based test procedures. Required at Level A only (§6.4.4.2.d).
- **Control Coupling**: Control flow interactions between code components are exercised by requirements-based test procedures. Required at Level A only (§6.4.4.2.e).

**Q7: What is MC/DC and how does it differ from decision coverage? Give an example.**
A: Decision coverage (§6.4.4.2.b) requires every Boolean expression to evaluate to both True and False at least once. MC/DC (§6.4.4.2.c) goes further: it requires that for each condition in a decision, there exist a test case where ONLY that condition changes value while all other conditions remain constant, and the decision outcome changes as a result. This demonstrates each condition INDEPENDENTLY affects the decision.

Example: Decision `A AND B`
- Decision coverage: need True (A=T, B=T) and False (A=F, B=F) → 2 test cases
- MC/DC: need to show A independently controls outcome: (A=T, B=T → True) and (A=F, B=T → False). And B independently controls: (A=T, B=T → True) and (A=T, B=F → False). With test set {(T,T), (F,T), (T,F)} = 3 test cases = N+1 where N=2 conditions.

**Masking**: occurs when short-circuit evaluation prevents a condition from affecting the outcome. For example, in `A OR B`, if A is True, B is never evaluated. MC/DC requires testing that each condition CAN affect the outcome — masking must be explicitly addressed.

**Q8: When structural coverage analysis reveals uncovered code, what are the possible resolutions?**
A: Per DO-178C §6.4.4.3, unexecuted code must be analyzed and resolved:
- **Shortcomings in test cases/procedures (§6.4.4.3.a)**: Test cases must be augmented or procedures changed. The requirements-based coverage analysis method may need review.
- **Inadequacies in software requirements (§6.4.4.3.b)**: Software requirements must be modified; additional test cases developed and procedures executed.
- **Extraneous code including dead code (§6.4.4.3.c)**: Code must be removed and analysis performed. May only remain if analysis shows it does NOT exist in the EOC (e.g., smart compiler/linker eliminates it), with procedures in place to prevent inclusion in future builds.
- **Deactivated code (§6.4.4.3.d)**: 
  - Category One (never executed in any certified configuration): combination of analysis and testing shows inadvertent execution is prevented; any software level reassignment justified by safety assessment and documented in PSAC
  - Category Two (executed in certain approved configurations): establish operational configuration; develop additional test cases

Any unexplained structural coverage gap is a certification finding.

**Q9: What is the difference between dead code and deactivated code?**
A: DO-178C distinguishes these explicitly. **Dead code** is code that is unreachable — it cannot be executed by any means regardless of configuration (§6.4.4.3.c). It has no legitimate purpose in the software and must be removed or justified if it somehow doesn't appear in the EOC. **Deactivated code** (§2.5.4, §5.2.4, §6.4.4.3.d) is intentionally inactive in specific configurations but IS executable code designed to be used in other approved configurations (option-selectable software). It must be formally designed: a mechanism must assure it has no adverse impact on active code; its development must meet DO-178C objectives just like active code; evidence must show it's disabled where not intended.

**Q10: What is the PSAC and what does it contain?**
A: The Plan for Software Aspects of Certification (§11.1) is the primary document submitted to the certification authority before development begins. It serves as the primary means for the authority to assess whether the proposed lifecycle is commensurate with the required software level rigor. The PSAC is agreed with the authority and DER at SOI-1 and forms the agreement for the certification program.

Content: system overview; software overview; certification considerations (certification basis, means of compliance, software level(s) with justification); software life cycle description (summary of each process, how objectives satisfied, organizations and responsibilities); software life cycle data (what data produced, relationship between data items, data submitted to authority, form and availability); schedule; additional considerations (alternative methods, tool qualification, PDS, option-selectable, user-modifiable, deactivated code, COTS, field-loadable, parameter data items, dissimilar software, product service history); supplier oversight.

**Q11: What is the Software Accomplishment Summary and how does it differ from the PSAC?**
A: The PSAC (§11.1) is submitted BEFORE development begins — it proposes what will be done and how compliance will be achieved. It is a forward-looking agreement. The SAS (§11.20) is submitted AT CERTIFICATION CLOSURE — it summarizes what was actually done and confirms all objectives were satisfied. It is backward-looking evidence. 

The SAS is the PRIMARY DATA ITEM for showing compliance with the PSAC. It must explain any differences between what was proposed in the PSAC and what was actually done. It includes the software characteristics (actual EOC size, timing margins, memory margins), the open Problem Report summary with justification for each open PR, and a formal compliance statement. Minimum data submitted to certification authority: PSAC + SCI + SAS (§9.3).

**Q12: Describe the tool qualification framework in DO-178C and what determines if a tool needs qualification.**
A: Per DO-178C §12.2, tool qualification is needed when a tool ELIMINATES, REDUCES, OR AUTOMATES lifecycle processes of DO-178C WITHOUT its output being verified as specified in §6. If the tool's output is independently verified by other means (a human reviews it), no qualification is needed.

The Tool Qualification Level (TQL) is determined from three criteria applied sequentially (Table 12-1):
- Criteria 1 (can INSERT error — development tool): TQL-1 at Level A, TQL-2 at B, TQL-3 at C, TQL-4 at D
- Criteria 2 (can FAIL TO DETECT error AND eliminates other verification): TQL-4 at A/B, TQL-5 at C/D
- Criteria 3 (can fail to detect error, errors caught by other means): TQL-5 at all levels

TQL-1 is most rigorous; TQL-5 is least. The detailed objectives for each TQL are defined in DO-330. A tool is qualified only for a specific system — requalification needed for another system. Intention to use the tool must be stated in the PSAC.

Practical example: VectorCAST used for structural coverage analysis at Level A. If coverage reports accepted without independent review → Criteria 2 → TQL-4. If an independent engineer manually reviews the coverage report → no qualification needed.

**Q13: What is the difference between CC1 and CC2 data control categories?**
A: DO-178C §7.3 defines two SCM control categories. **CC1** is the full set of SCM activities: configuration identification, baselines, traceability, problem reporting, full change control (tracking, records, approval), change review, configuration status accounting, full archival including retrieval, protection against unauthorized changes, media selection, release, and data retention. **CC2** is a subset: configuration identification only, retrieval, protection against unauthorized changes, and data retention. CC2 data is NOT expected to change incrementally but will be superseded by new data.

In practice: safety-critical lifecycle data (Source Code, EOC, test procedures, plans) gets CC1 at higher software levels. Supporting data may get CC2. The specific category for each data item at each software level is specified in the Annex A tables.

**Q14: When does software verification need to be reverified after a change?**
A: DO-178C requires reverification (§6.2 and §7.2.4) when changes are made that could impact previously verified functionality. The change control activity (§7.2.4) requires: changes traced to their origin; lifecycle processes repeated from the point at which the change affects their outputs. Example: an error discovered at HW/SW integration shown to result from an incorrect design → design correction, code correction, AND repetition of all associated integral process activities (reviews, tests covering the changed areas). Reverification scope is determined by the impact analysis — all affected areas must be reverified, not just the changed module.

**Q15: Explain the concept of robustness testing in DO-178C.**
A: Robustness testing (§6.4.2.2) demonstrates that the software responds correctly to ABNORMAL inputs and conditions. It is requirements-based — the software requirements must specify the correct response to abnormal conditions and inputs. If requirements are complete for abnormal conditions, robustness test cases follow from those requirements. If requirements do not cover abnormal conditions, the test cases may reveal requirements inadequacies requiring requirements update.

Robustness test activities include: exercising real/integer variables with invalid equivalence class values; testing system initialization during abnormal conditions; determining failure modes of incoming data (complex digital strings from external systems); attempting out-of-range computed loop count values; testing protection mechanisms for exceeded frame times; testing arithmetic overflow protection for time-related functions (filters, integrators, delays); provoking state transitions NOT allowed by requirements.

**Q16: What is data coupling analysis and why is it only required at Level A?**
A: Data coupling (§6.4.4.2.d) is a structural coverage criterion that confirms requirements-based testing has exercised the DATA INTERACTIONS between code components at their interfaces — passed parameters, global variables, shared data structures, interface registers. It addresses the risk that errors at software component interfaces (incorrect data types, range violations, timing issues in shared data access) might not be revealed by unit-level coverage alone.

It is only required at Level A because: at Level A, any undetected error could contribute to a catastrophic failure condition. The additional rigor of verifying not just statement/decision/MC/DC within individual components but also that the DATA CONNECTIONS between components have been exercised provides the highest confidence. At Levels B and C, the combination of requirements-based testing, decision/MC/DC coverage, and independent reviews is deemed sufficient.

**Q17: What is control coupling and how does it differ from data coupling?**
A: Control coupling (§6.4.4.2.e) is a structural coverage criterion that confirms requirements-based testing has exercised CONTROL FLOW INTERACTIONS between code components — the sequencing and conditionality of calls, function invocations, event notifications, and exception handling between components.

Difference: Data coupling focuses on DATA passing between components (parameters, global variables). Control coupling focuses on CONTROL passing between components — whether component A calls component B correctly, whether callbacks are invoked in the right sequence, whether control returns properly. Both are required only at Level A. Together they address the interface errors between components that might not be detectable through component-level coverage alone.

**Q18: What are the three types of software testing in DO-178C and what errors does each typically reveal?**
A: Per DO-178C §6.4.3:

**HW/SW Integration Testing (§6.4.3.a)**: Concentrates on errors associated with the software operating within the target computer environment and high-level functionality. Typical errors: incorrect interrupt handling; failure to satisfy execution time requirements; incorrect response to hardware transients or hardware failures (startup sequencing, transient input loads, power transients); data bus and resource contention (memory mapping); inability of built-in test to detect failures; HW/SW interface errors; incorrect control loop behavior; incorrect control of memory management hardware; stack overflow; incorrect operation of field-loadable software compatibility checking; violations of software partitioning.

**Software Integration Testing (§6.4.3.b)**: Concentrates on inter-relationships between SW requirements and implementation by architecture. Typical errors: incorrect initialization of variables and constants; parameter passing errors; data corruption (especially global data); inadequate end-to-end numerical resolution; incorrect sequencing of events and operations.

**Low-Level Testing (§6.4.3.c)**: Concentrates on each software component's compliance with its LLRs. Typical errors: algorithm failure; incorrect loop operations; incorrect logic decisions; failure to process legitimate input combinations; incorrect responses to missing/corrupted input data; incorrect handling of exceptions (arithmetic faults, array limit violations); incorrect computation sequence; inadequate algorithm precision, accuracy, or performance.

**Q19: Can you use a test performed for HW/SW integration to satisfy both HW/SW integration testing and structural coverage objectives?**
A: Yes, per the Note at §6.4: "If a test case and its corresponding test procedure are developed and executed for hardware/software integration testing or software integration testing, and satisfy the requirements-based coverage and structural coverage, it is not necessary to duplicate the test for low-level testing." 

The same test can count toward multiple coverage objectives as long as it satisfies both the requirements-based coverage criteria (traced to requirements) AND the structural coverage criteria. However, the Note warns: "Substituting nominally equivalent low-level tests for high-level tests may be less effective due to the reduced amount of overall functionality tested." The preference is to perform HW/SW integration tests that also achieve structural coverage, rather than running separate low-level unit tests.

**Q20: What is the problem reporting lifecycle and what are the key requirements for a problem report?**
A: Problem Reports (§7.2.3, §11.17) capture process non-compliance with plans and standards, deficiencies of lifecycle process outputs, and anomalous behavior of software products. They are not just for software bugs — they apply to any lifecycle data output deficiency.

A Problem Report must include (§11.17): identification of the CI and/or lifecycle process activity where the problem was observed; identification of CI(s) to be modified or process to be changed; problem description sufficient to understand and resolve the problem with enough detail to assess potential safety/functional effects; description of corrective action taken.

Problem Reports that require corrective action of the software product MUST invoke the change control activity (§7.2.3.c). Open Problem Reports at certification must be evaluated and summarized in the SAS (§11.20.k) with: description, associated errors, functional limitations, operational restrictions, potential adverse safety effects, justification for remaining open, and mitigation actions. A safety-significant open PR is a certification finding.
